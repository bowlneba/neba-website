# v0.1 Public Release Polish Plan

## Context

The public-facing site is feature-complete (tournaments, stats, news, sponsors, history, HoF, documents, bowling centers, contact). No admin UI exists yet. Before tagging v0.1, this plan addresses security gaps, SEO/discoverability, test coverage holes, and minor code polish — all scoped strictly to the public view.

---

## P0 — Must Fix Before Launch

### 1. Strip sensitive sponsor fields from anonymous API response

**Problem:** `GET /sponsors/{slug}` is `AllowAnonymous`, but the response (`SponsorDetailResponse` in Contracts) includes `PromotionalNotes`, `LiveReadText`, `SponsorContactName`, `SponsorContactEmailAddress`, `SponsorContactPhoneNumber`, and `SponsorContactPhoneNumberType`. The Blazor page conditionally renders these when `HasPromotionalInfo == true`, but anyone who calls the API directly gets the raw JSON with these fields populated.

These are member-only data fields (live-read scripts, internal NEBA contact info, promo notes) that should never be public.

**Fix:** Remove those six fields from:
- `src/Neba.Api.Contracts/Sponsors/SponsorDetailResponse.cs`
- `src/Neba.Api/Features/Sponsors/GetSponsorDetail/SponsorDetailDto.cs`
- `src/Neba.Api/Features/Sponsors/GetSponsorDetail/GetSponsorDetailQueryHandler.cs` (drop from projection + DTO construction)
- `src/Neba.Api/Features/Sponsors/GetSponsorDetail/GetSponsorDetailEndpoint.cs` (drop from response mapping)
- `src/Neba.Api/Features/Sponsors/GetSponsorDetail/GetSponsorDetailEndpointSummary.cs` (example response)
- `src/Neba.Website.Server/Sponsors/SponsorDetail.razor` — remove the two `HasPromotionalInfo` and `HasInternalContact` sections entirely (the data won't be there anymore), remove the `TODO` comments
- `src/Neba.Website.Server/Sponsors/SponsorDetailViewModel.cs` — remove corresponding view-model properties
- Update/delete related tests in `Neba.Api.Tests` and `Neba.Website.Tests`

v0.2 adds authentication; at that point a new authenticated-only endpoint (or header-gated response) will return these fields.

## P1 — High Priority Polish (do for v0.1)

### 2. SEO: robots.txt + sitemap.xml

**Problem:** Neither file exists. Search engines will crawl without guidance; the sitemap omission slows indexing.

**Fix:**
- Add `wwwroot/robots.txt` to `Neba.Website.Server`:
  ```
  User-agent: *
  Allow: /
  Sitemap: https://www.bowlneba.com/sitemap.xml
  ```
- Add `wwwroot/sitemap.xml` listing all static public routes (`/`, `/about`, `/tournaments`, `/stats`, `/history/champions`, `/history/awards/*`, `/hall-of-fame`, `/sponsors`, `/news`, `/bowling-centers`, `/bylaws`, `/tournament-rules`, `/contact`) with `<lastmod>` and `<changefreq>` hints.

  Dynamic routes (individual tournaments, sponsor detail, news articles) are harder — for v0.1 it's acceptable to omit them and add a dynamic sitemap endpoint in v0.2.

### 3. SEO: Open Graph meta tags on key pages

**Problem:** No page sets `og:title`, `og:description`, `og:image`, or `twitter:card`. Social shares and link previews will render bare.

**Fix:** Add `<HeadContent>` OG tags to the five highest-traffic pages:
- `Pages/Home.razor` — og:image = neba-logo.png
- `Tournaments/Schedule/Tournaments.razor`
- `News/NewsList.razor` and `News/NewsDetail.razor` (dynamic title/description from article)
- `HallOfFame/HallOfFame.razor`

Pattern already in use for `<meta name="description">` — OG tags are an additive change alongside it.

### 4. E2E test gaps

**Problem:** No E2E spec files for Home or Contact. Every other public page has a spec.

**Fix:** Add two new spec files:
- `tests/e2e/Home.spec.ts` — verify page title, hero heading, motto cards, quick stats, quick-link nav cards
- `tests/e2e/Contact.spec.ts` — verify page title, three department cards (General/Tournament/Website), email links are present

Follow existing fixture pattern from `tests/e2e/About.spec.ts` (static pages that don't need API fixtures).

## P2 — v0.1 Completeness

### 5. API rate limiting

**Problem:** No rate limiting on any endpoint. The API is public and has no throttle protection.

**Fix:** Add ASP.NET Core built-in rate limiting in `Neba.Api/Program.cs`:
```csharp
builder.Services.AddRateLimiter(options =>
{
    options.AddFixedWindowLimiter("public", o =>
    {
        o.PermitLimit = 100;
        o.Window = TimeSpan.FromMinutes(1);
        o.QueueProcessingOrder = QueueProcessingOrder.OldestFirst;
        o.QueueLimit = 0;
    });
    options.RejectionStatusCode = StatusCodes.Status429TooManyRequests;
});
```
Apply the policy to the FastEndpoints group (or globally). The website server calls the API on behalf of users, so the effective per-IP limit is already much lower from end-user perspective.

### 6. Architecture tests

**Problem:** PR guidelines enforce cross-domain boundaries, command/query return type rules, etc. — but these are verified by human review only. A regression could silently pass CI.

**Fix:** Add `tests/Neba.Architecture.Tests` project using `NetArchTest.Rules`:
- No type in `Features/*/Domain/` imports from another feature's domain (except strongly-typed IDs)
- Command handlers return `ErrorOr<T>`
- Query handlers don't return domain entity types
- All endpoints have an explicit auth declaration (`AllowAnonymous`, `Roles`, or `Policies`)

## P3 — Defer to v0.2

- **Auth gating for member content** — sponsor promotional data, live-read scripts, internal contacts. P0 above removes the data from the public response; v0.2 adds it back behind auth.
- **Dynamic sitemap** — endpoint that generates XML for tournaments, news articles, sponsor slugs.
- **DebugCacheTools guard** — currently `#if DEBUG` compile-time only. Fine for production Release builds; consider adding a runtime `IsDevelopment()` guard as a belt-and-suspenders check.
- **HybridCache in-memory layer** — already deferred per docs.
